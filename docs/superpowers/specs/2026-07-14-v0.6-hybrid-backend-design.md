# GEODESIC-M v0.6 — Hybrid / GPU Transfer Optimization (M3): Design Spec

- Date: 2026-07-14
- Author: Felipe Carvajal Brown
- Milestone: M3 (SAD.md §7.4), ROADMAP.md v0.6
- Status: Draft — awaiting Felipe's approval before an implementation plan is written
- Architecture reference: SAD.md §7.2, §7.3, §7.4, §2.3, §2.4, §8

---

## 1. Summary

v0.6 makes the GPU backend transfer-efficient without changing any physics, any
precision, or any public interface. The v0.5 `GpuBackend` already implements the
SAD.md §7.4 workload split — non-bonded LJ on the GPU, bonded forces plus the
constraint solve plus the neighbor rebuild on the CPU in f64. What it lacks is
transfer management: `NonbondedKernel::evaluate` reallocates every GPU buffer on
every `compute_forces` call and re-uploads data that never changes between steps
(sigma, epsilon) or changes only on rebuild (the CSR neighbor list). v0.6
replaces that per-step allocation churn with GPU buffers that live for the
lifetime of the run, uploading only what genuinely changes each step.

This is the whole milestone. The `ComputeBackend` trait does not change, the
BAB + RATTLE run loop in `geodesic/src/lib.rs` does not change, and the CPU f64
integration path is untouched. The optimization is entirely internal to
`geodesic-gpu`.

A second, smaller deliverable: `backend = "hybrid"` in `config.toml` stops
returning an error and becomes a documented alias that resolves to the same
optimized GPU path under the `gpu` feature.

---

## 2. Context — where v0.5 left off

The v0.5 `GpuBackend` (`geodesic-gpu/src/gpu_backend.rs`) is already the §7.4
hybrid: `compute_forces` runs bonded bond/angle/dihedral forces on the CPU
(`bonded::compute_*`), dispatches the non-bonded LJ loop to the GPU
(`kernel.evaluate`), and `geodesic_drift` runs the SHAKE/RATTLE constraint solve
on the CPU (`integrator::drift_and_constrain`). `build_neighbor_list` rebuilds
the Verlet list on the CPU and expands the half pair-list into a per-atom CSR
gather list (`build_csr`). The naming split the ROADMAP drew between "pure GPU"
(v0.5) and "hybrid" (v0.6) does not correspond to two different engines — it is
one engine, and v0.5 already built the workload split.

What v0.5 did not do — deliberately, because v0.5 was correctness-first — is
manage transfers. Every call to `NonbondedKernel::evaluate`
(`geodesic-gpu/src/kernel.rs:91`) does the following, from scratch, on every
step:

- Allocates a fresh positions buffer, converting f64 → f32 and packing to
  `vec4<f32>`.
- Allocates fresh sigma and epsilon buffers, re-uploading values that never
  change during a run.
- Allocates fresh offsets and neighbors buffers, re-uploading a CSR that only
  changes on a neighbor rebuild.
- Allocates a fresh output buffer and a fresh readback buffer.
- Creates a fresh bind group tying all seven bindings together.

### 2.1 The framing constraint (why "resident positions" cannot go further)

The obvious ambition — keep positions resident on the device across steps and
never round-trip them — is not reachable while integration stays on the CPU.
The constraint solve is on the CPU (SAD.md §7.4 mandates this: iterative,
convergence-dependent). So the host produces new positions every single step
inside `drift_and_constrain`. Those new positions must go host → device before
the next force evaluation. The per-step position upload is therefore
unavoidable, and so is the per-step device → host force readback (the CPU
`half_kick` needs the forces immediately). SAD.md §7.4's line "positions and
velocities live on GPU during force computation; copied to CPU before the
constraint solve step" describes exactly this: they are resident for the
duration of the kernel dispatch, not across the CPU constraint solve.

What v0.6 can eliminate is everything else: the per-step reallocation of every
buffer, and the per-step re-upload of static data (sigma, epsilon) and
rebuild-frequency data (the CSR). That is the scope, stated honestly.

## 3. Goals

- G1. Persistent GPU buffers: allocate the non-bonded kernel's buffers once and
  reuse them across all steps, sizing them to `n_atoms` at construction.
- G2. Upload only what changed: positions every step (unavoidable), sigma and
  epsilon once at construction, the CSR only on a neighbor rebuild.
- G3. No change to observable behavior: GPU and CPU forces still agree at the
  v0.5 f32 tolerance; two GPU evaluations on the same adapter are still
  bit-identical (ADR 0003 holds).
- G4. `backend = "hybrid"` resolves and runs, bit-identical to `backend = "gpu"`.
- G5. A multi-step trajectory test asserting the GPU run matches the CPU run
  (energy CSV and final state) within the v0.5 f32 tolerance over N steps on
  `ala_dipeptide` — this encodes the ROADMAP v0.6 exit criterion.

---

## 4. Non-goals

- N1. No change to the `ComputeBackend` trait or the run loop. The optimization
  is internal to `geodesic-gpu`.
- N2. No change to precision. CPU integration (half_kick, drift, SHAKE, RATTLE,
  velocity-constrain) stays in f64; the GPU non-bonded loop stays f32 (ADR 0002).
- N3. No GPU-resident integration. The constraint solve stays on the CPU per
  §7.4; positions round-trip host ↔ device every step by necessity (§2.1).
- N4. No minimum-image in the bonded/constraint terms. Deferred again — see §12.
- N5. No GPU constraint convergence-reduce. Dropped — see ADR 0006 (§10.2) and
  §12.
- N6. No new WGSL kernel and no change to the existing `nonbonded.wgsl`. The
  kernel's math, bindings, and workgroup layout are unchanged; only how its
  buffers are allocated and fed changes.

## 5. Approved design decisions (the four forks)

All four were decided with Felipe in the prior brainstorming session and are not
re-opened here. They are recorded for traceability.

### 5.1 Ambition: transfer-efficient, CPU integration stays f64

Persistent GPU buffers. All of half_kick / drift / SHAKE / RATTLE /
velocity-constrain stay on the CPU in f64. No change to the `ComputeBackend`
trait or the run loop — the optimization is entirely internal to the backend.

### 5.2 Backend identity: optimize `GpuBackend` in place; `Hybrid` is an alias

There is no distinct pure-GPU vs hybrid engine in GEODESIC-M — v0.5's GPU
backend already is the §7.4 split. So v0.6 optimizes `GpuBackend` in place, and
`Backend::Hybrid` becomes a documented alias that resolves to the same optimized
`GpuBackend` path under the `gpu` feature (and the same actionable "rebuild with
--features gpu" error when the feature is absent). The ROADMAP's three-way exit
criterion (hybrid matches pure-CPU and pure-GPU) collapses to CPU ≈ GPU, because
"hybrid" and "gpu" are the same code path. Recorded as ADR 0005.

### 5.3 Minimum-image in bonded/constraint terms: defer again

Non-goal for v0.6. Every v0.6 exit fixture is non-periodic (`periodic = false`),
where the min-image term is a no-op, and no periodic bonded fixture exists to
test it against. Noted as a spec non-goal (§12); no new ADR, since it was
already deferred through v0.4 and v0.5.

### 5.4 GPU constraint convergence-reduce: drop via ADR

Dropped. With the constraint solve on the CPU (§7.4), the lambdas are already on
the host, so a GPU `max|lambda|` reduce would be a net-negative transfer: it
would ship data to the device only to read a scalar back. It only pays off if
the whole solve runs on the GPU, which §7.4 forbids. This resolves a standing
contradiction between SAD.md §7.3 (which describes a GPU convergence-reduce) and
§7.4 (which keeps the solve on the CPU). Recorded as ADR 0006.

## 6. The mechanism — persistent buffers

The change lives in `geodesic-gpu`, principally in the non-bonded kernel
(`kernel.rs`) and its use by `GpuBackend` (`gpu_backend.rs`). At a design level:
the kernel gains ownership of its GPU buffers and its bind group, is constructed
knowing `n_atoms`, exposes a one-time static-data upload, and turns `evaluate`
into an update-then-dispatch that writes only changed data via
`queue.write_buffer` into buffers that already exist.

### 6.1 Buffer inventory and lifetimes

| Buffer | Element / size | Written | Reallocated | wgpu usage |
| :--- | :--- | :--- | :--- | :--- |
| positions | `vec4<f32>` × n | every `compute_forces` | never | STORAGE \| COPY_DST |
| sigma | `f32` × n | once, at construction | never | STORAGE \| COPY_DST |
| epsilon | `f32` × n | once, at construction | never | STORAGE \| COPY_DST |
| offsets | `u32` × (n+1) | every `build_neighbor_list` | never (fixed size) | STORAGE \| COPY_DST |
| neighbors | `u32` × len(CSR) | every `build_neighbor_list` | only when a rebuild is longer than current capacity | STORAGE \| COPY_DST |
| out_force | `vec4<f32>` × n | GPU writes each dispatch | never | STORAGE \| COPY_SRC |
| readback | `vec4<f32>` × n | GPU copy target each dispatch | never | MAP_READ \| COPY_DST |
| params (uniform) | `ParamsUniform` | once (or on the rare cutoff change) | never | UNIFORM \| COPY_DST |

The `COPY_DST` usage flag is new and required: v0.5 creates these with
`create_buffer_init` (STORAGE only) because it never writes them again;
persistent buffers must be writable via `queue.write_buffer`, which requires
`COPY_DST`.

`ParamsUniform` (n_atoms, r_cutoff, r_switch, box_size) is constant for a run in
the current design, so it is written once. It is listed as `COPY_DST` only to
keep the door open cheaply; it is not rewritten per step.

### 6.2 Construction (`NonbondedKernel::new` / `GpuBackend::try_new`)

At construction, `n_atoms` is known from `AtomData`. The kernel allocates every
buffer in §6.1 at its final size (except neighbors, which starts at a
zero/placeholder capacity and grows on first rebuild). sigma and epsilon are
uploaded once here, from `atoms.sigma` / `atoms.epsilon` cast to f32. The params
uniform is uploaded here. The bind group is created here, referencing the
current neighbors buffer.

### 6.3 Per-rebuild (`GpuBackend::build_neighbor_list`)

Unchanged on the CPU side: `neighbor::build` produces the Verlet list and
`build_csr` expands it to `(offsets, neighbors)`. New GPU-side handling:

- offsets: fixed size n+1, always written in place with `queue.write_buffer`.
- neighbors: compare the new CSR length against the tracked capacity. If it
  fits, write in place. If it is longer, reallocate the neighbors buffer to the
  new length (optionally with headroom), then recreate the bind group — this is
  the only condition that triggers a bind-group rebuild.

### 6.4 Per-step (`GpuBackend::compute_forces` → `NonbondedKernel::evaluate`)

1. CPU bonded forces run first into `reduced.{fx,fy,fz}`, exactly as in v0.5.
2. Positions are packed f64 → f32 `vec4<f32>` and written into the existing
   positions buffer via `queue.write_buffer` — no allocation, no bind-group
   change (same buffer handle).
3. The compute pass is encoded and dispatched against the persistent bind group
   (`ceil(n/64)` workgroups, as today).
4. `out_force` is copied to `readback`; `map_async` + `poll(Wait)` reads the
   forces and the per-atom energy back. This per-step readback is the one
   unavoidable sync, unchanged from v0.5.
5. Forces are added into `reduced` and the halved non-bonded energy into
   `potential_energy`, exactly as today.

### 6.5 Bind group management

The bind group is created once at construction and recreated only when the
neighbors buffer reallocates (§6.3). Position writes, sigma/epsilon writes, and
offsets writes all target buffers whose handles do not change, so they never
invalidate the bind group. Between two neighbor rebuilds, every step reuses one
bind group.

### 6.6 What stays exactly the same

- `nonbonded.wgsl`: identical. Same LJ math, same quintic switching function,
  same min-image on the non-bonded pair, same one-thread-per-atom CSR gather,
  same `vec4<f32>` layout, same `@workgroup_size(64)`.
- The CPU path in `compute_forces` (bonded forces) and `geodesic_drift` (SHAKE +
  RATTLE): identical.
- The `0.5 * energy` double-count correction on the non-bonded energy: identical.
- Determinism construction (no atomics, fixed-order CSR gather): identical.

## 7. Backend identity — the `Hybrid` alias

Today `run()` in `geodesic/src/lib.rs` matches `Backend::Hybrid` to an error
("the hybrid backend lands in v0.6"). v0.6 changes that arm to resolve `Hybrid`
to the same construction as `Gpu`:

- Under `--features gpu`: `Backend::Hybrid` constructs `GpuBackend::try_new`,
  the same as `Backend::Gpu`.
- Without the feature: `Backend::Hybrid` returns the same actionable
  "compiled without the `gpu` feature; rebuild with --features gpu" error that
  `Backend::Gpu` returns, keyed to `run.backend`.

`config.toml`'s `backend` field documents `"hybrid"` as an alias for `"gpu"`.
No new enum variant behavior, no second backend struct.

---

## 8. Determinism argument

ADR 0003's guarantee — two evaluations on the same adapter are bit-identical —
must still hold with persistent buffers. It does, because:

- Positions are fully overwritten each step via `queue.write_buffer` before the
  dispatch that reads them; no stale position data survives into a step.
- The CSR (offsets + neighbors) is fully overwritten on each rebuild before the
  first dispatch that uses it; between rebuilds it is constant by construction.
- sigma and epsilon are constant for the run.
- The kernel is unchanged: one thread per atom, fixed-order gather over its CSR
  slice, no `atomicAdd`, fixed-order energy reduction.

Reusing a buffer does not change the bytes the kernel reads on any given step —
only when those bytes were uploaded. So the same inputs still produce the same
outputs, bit for bit, on the same adapter and driver.

---

## 9. Interfaces that do not change

- `ComputeBackend` trait (`geodesic-core`): unchanged. No new methods, no
  signature changes.
- The BAB + RATTLE run loop (`geodesic/src/lib.rs`): unchanged. It still holds a
  `Box<dyn ComputeBackend>` and drives build_neighbor_list → compute_forces →
  geodesic_drift blind to the concrete backend.
- CPU integration precision: f64 throughout, unchanged.
- `nonbonded.wgsl`: unchanged.

The only edits outside `geodesic-gpu` are: the `Backend::Hybrid` match arm in
`geodesic/src/lib.rs` (§7), the `backend` field doc in the config schema, and
new tests.

## 10. New ADRs

Two ADRs are added under `docs/adr/`, following the existing MADR-lite format
(Status / Date / Deciders: Felipe Carvajal Brown / Context / Decision /
Consequences / Alternatives considered), with rows added to
`docs/adr/README.md`. Their full proposed text is embedded below so the wording
can be reviewed alongside this spec; the files themselves are created as tasks in
the implementation plan, not in this spec-writing session.

### 10.1 ADR 0005 — Hybrid backend is a documented alias for the GPU backend

> Status: Accepted
> Date: 2026-07-14
> Deciders: Felipe Carvajal Brown
>
> **Context.** SAD.md §7.3 (M2, GPU) and §7.4 (M3, hybrid) read as two
> milestones producing two backends. In practice the v0.5 `GpuBackend` already
> implements the §7.4 workload split: non-bonded LJ on the GPU, bonded forces +
> constraint solve + neighbor rebuild on the CPU in f64. There is no second,
> distinct hybrid engine to build — only transfer efficiency to add to the one
> that exists. The `config.toml` `backend` field already accepts `"hybrid"`,
> which v0.5 rejects with an actionable "lands in v0.6" error.
>
> **Decision.** `backend = "hybrid"` is a documented alias for `backend = "gpu"`.
> Under `--features gpu` both construct the same optimized `GpuBackend`; without
> the feature both return the same actionable "rebuild with --features gpu"
> error keyed to `run.backend`. GEODESIC-M ships one GPU/hybrid backend, not two.
> The ROADMAP v0.6 exit criterion "hybrid matches pure-CPU and pure-GPU"
> collapses to "CPU ≈ GPU" because hybrid and gpu are the same code path.
>
> **Consequences.**
> - Users can select either name; behavior and output are bit-identical between
>   them on the same adapter.
> - One backend implementation to test and maintain, not two.
> - The SAD's M2/M3 milestone framing is documented here as one engine delivered
>   in two increments (v0.5 correctness, v0.6 transfer efficiency), not two
>   engines.
>
> **Alternatives considered.**
> - *A separate `HybridBackend` struct.* Rejected: it would duplicate the v0.5
>   backend with no behavioral difference, since v0.5 is already the §7.4 split.
> - *Keep rejecting `"hybrid"`.* Rejected: the workload split it names exists and
>   ships; rejecting the name is misleading.

### 10.2 ADR 0006 — GPU constraint convergence-reduce dropped

> Status: Accepted
> Date: 2026-07-14
> Deciders: Felipe Carvajal Brown
>
> **Context.** SAD.md §7.3 specifies a GPU tree reduction for the constraint
> solver's convergence check (global `max|lambda_i|`), "no CPU round-trip per
> iteration." SAD.md §7.4, the hybrid split, keeps the entire constraint solve on
> the CPU (iterative, convergence-dependent, CPU overhead acceptable). These two
> sections contradict each other: a GPU convergence-reduce only makes sense if
> the lambdas live on the GPU, but §7.4 computes them on the CPU.
>
> **Decision.** The GPU constraint convergence-reduce is dropped. With the solve
> on the CPU (SHAKE/RATTLE in `integrator::drift_and_constrain`), the lambdas and
> their residuals are already in host memory; the convergence check is a CPU
> reduction, as it is today. Shipping residuals to the GPU to reduce a single
> scalar and read it back would be a net-negative transfer.
>
> **Consequences.**
> - Resolves the §7.3 / §7.4 contradiction in favor of §7.4 (solve on CPU).
> - No GPU code is written for the constraint solver in v0.6; the convergence
>   check stays the deterministic CPU reduction already in the engine.
> - If a future milestone moves the whole constraint solve onto the GPU, a
>   GPU-side convergence-reduce can be reconsidered — it only pays off then.
>
> **Alternatives considered.**
> - *Implement the §7.3 GPU reduce as written.* Rejected: net-negative transfer
>   while the solve is on the CPU; adds a round-trip to read one scalar.
> - *Move the whole constraint solve to the GPU.* Rejected for v0.6: §7.4 forbids
>   it (iterative, convergence-dependent, poor GPU fit), and it is a far larger
>   scope than a transfer-optimization milestone.

## 11. Testing and exit criteria

### 11.1 Existing v0.5 GPU tests stay green

The v0.5 adapter-adaptive GPU tests continue to pass, now guarding the
persistent-buffer path:

- GPU vs CPU non-bonded forces agree at 1e-4 relative (f32) on `lj_pair`,
  `water_box_4`, and the exclusion-heavy `ala_dipeptide`.
- Two evaluations on the same adapter are bit-identical (`Vec<[f32;3]>` and
  energy bits) — ADR 0003 under the new buffer reuse.

### 11.2 New: multi-step trajectory equivalence (the exit criterion)

Run the full BAB + RATTLE loop with `backend = gpu` and assert the energy CSV
and the final `SimState` match the CPU run within the v0.5 f32 tolerance over N
steps on `ala_dipeptide`. This is the ROADMAP v0.6 exit criterion, made
executable. Adapter-adaptive: skip-with-log when no DX12/Vulkan adapter is
present, consistent with the v0.5 GPU tests.

### 11.3 New: alias guard

`backend = "hybrid"` resolves and runs, and its output is bit-identical to
`backend = "gpu"` on the same adapter.

### 11.4 Optional (not exit-blocking): full-step benchmark

`bench_full_step` under `--features gpu` to capture the speedup of the
persistent-buffer path against the v0.5 per-step-allocation baseline. Numeric
baselines are hardware-specific and captured per-runner; this benchmark
documents the win, it does not gate the release.

### 11.5 Exit criteria (ROADMAP v0.6), restated

The hybrid/GPU backend matches the pure-CPU energy trajectory on the fixture
suite within the v0.5 tolerance, with the three-way criterion collapsed to
CPU ≈ GPU per ADR 0005. `cargo build --features gpu` and
`cargo clippy --features gpu -- -D warnings` stay green on Windows and Linux CI.

## 12. Deferred / out of scope

- **Minimum-image in bonded and constraint terms (SAD.md §2.4).** Deferred
  again, consistent with v0.4 and v0.5. Every v0.6 exit fixture is non-periodic
  (`periodic = false`), where the term is a no-op, and there is no periodic
  bonded fixture to validate it. Do it when a genuinely periodic bonded
  production system is first run. No ADR — it is a standing deferral, not a new
  decision.
- **GPU-resident integration.** Out of scope by §7.4: the constraint solve stays
  on the CPU, so positions round-trip every step. Not revisited in v0.6.
- **GPU constraint convergence-reduce.** Dropped, ADR 0006.
- **The O(N²) neighbor build.** A known perf target (memory.md), unrelated to
  transfer optimization; not part of v0.6.

## 13. Files touched (design-level, for orientation only)

Concrete task ordering is the implementation plan's job. At a glance, v0.6
touches:

- `geodesic-gpu/src/kernel.rs` — persistent buffers, one-time static upload,
  `evaluate` becomes update-then-dispatch. (Primary change.)
- `geodesic-gpu/src/gpu_backend.rs` — construct persistent buffers with
  `n_atoms`; `build_neighbor_list` writes offsets in place and grows neighbors
  only when needed.
- `geodesic/src/lib.rs` — `Backend::Hybrid` match arm resolves to the GPU path.
- `geodesic-io` config schema doc — `"hybrid"` documented as an alias.
- `docs/adr/0005-*.md`, `docs/adr/0006-*.md`, `docs/adr/README.md` — new ADRs.
- Tests: multi-step trajectory equivalence, alias guard; optional
  `bench_full_step` under `--features gpu`.
- `nonbonded.wgsl` — not touched.
